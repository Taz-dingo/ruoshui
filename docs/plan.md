# 项目 Plan

## 阶段 0：项目定义

目标：统一方向，固定第一版边界。

交付物：

- `README.md`
- `docs/spec.md`
- `docs/plan.md`
- `docs/tasks.md`

完成标准：

- 对项目目标、范围、阶段边界有一致理解
- 明确第一版为 `Web` + 桌面端 + 空中版 `3DGS` 纪念展示

## 阶段 1：资产可行性验证

目标：验证 `assets/raw` 中现有俯视五向图能否支撑空中版 `3DGS`。

任务：

- 整理原始素材目录结构
- 统计图片数量、分辨率、文件体积
- 记录素材批次、覆盖范围与明显缺失区域
- 选定首个最小重建流程
- 跑通一次可评估的 `3DGS` 实验
- 记录结果质量、问题区域、资产体积

产出：

- 素材盘点文档
- 重建实验记录
- 是否继续坚持当前素材方案的结论

完成标准：

- 至少得到一版可评估结果
- 明确主要缺陷来自素材、训练还是导出链路
- 明确原始 splat 是否有机会用于 `Web`

## 阶段 2：Web 展示原型

目标：把场景放进网页并能在桌面端被浏览。

任务：

- 初始化前端项目
- 选定最合适的 `3DGS` viewer 集成方案
- 搭建首页与场景页基础框架
- 集成场景加载、错误态、空状态
- 支持 `2-4` 个预设镜头

产出：

- 可本地运行的桌面端原型站点

完成标准：

- 本地可打开页面并完成场景浏览
- 首次进入时能感受到基本的纪念展示体验

## 阶段 3：叙事与热点内容

目标：让项目从“技术展示”变成“纪念展示”。

任务：

- 定义点位内容结构
- 选择若干代表性地点
- 接入图文内容面板
- 补足首页文案与项目缘起
- 完成纪念向、回忆向的基础视觉风格

产出：

- 一版可演示的数字纪念馆

完成标准：

- 场景浏览和内容阅读形成闭环
- 用户可以理解“若水广场”这个项目在纪念什么

## 阶段 4：性能与发布

目标：让项目可以稳定对外访问。

任务：

- 压缩场景资产
- 优化加载策略
- 优化桌面端体验
- 选择并完成部署
- 视情况补基础移动端兼容

产出：

- 线上可访问版本

完成标准：

- 桌面端可打开、可浏览、可分享
- 场景加载时间与资源体积处于可接受范围

## 阶段 5：第二阶段能力（暂缓）

候选内容：

- 后台管理
- 登录与权限
- 用户投稿
- 视频内容
- 小程序版本
- 更丰富的内容编辑能力

## 迭代节奏建议

每次迭代遵循以下模板：

1. 先写本次目标
2. 定义完成标准
3. 只改动相关文件
4. 做最小验证
5. 记录结果与下一步

推荐单次迭代大小：

- `30` 分钟到 `2` 小时
- 只解决一个问题
- 最终产出必须“可看”或“可判断”
- 优先做能尽快暴露风险的工作
