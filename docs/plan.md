# 项目 Plan

## 阶段 0：项目定义

目标：统一方向，固定第一版边界。

交付物：

- `README.md`
- `docs/spec.md`
- `docs/plan.md`
- `docs/tasks.md`

完成标准：

- 对项目目标、范围、阶段边界有一致理解

## 阶段 1：资产可行性验证

目标：验证现有航拍五向图能否支撑空中版 3DGS。

任务：

- 整理原始素材目录结构
- 统计图片数量、分辨率、覆盖角度
- 跑通一次最小重建流程
- 记录结果质量、问题区域、资产体积

产出：

- 素材报告
- 重建实验记录
- 第一版是否继续坚持 3DGS 的结论

完成标准：

- 至少得到一版可评估结果
- 明确主要缺陷来自素材、训练还是导出链路

## 阶段 2：Web 展示原型

目标：把场景放进网页并能被浏览。

任务：

- 初始化前端项目
- 搭建基础页面框架
- 集成 3DGS viewer
- 做加载态、错误态、空状态
- 支持 2-4 个预设镜头

产出：

- 可本地运行的原型站点

完成标准：

- 本地可打开页面并完成场景浏览

## 阶段 3：叙事与热点内容

目标：让项目从“技术展示”变成“纪念项目”。

任务：

- 定义点位内容结构
- 配置若干代表性地点
- 接入图文内容面板
- 完成基础 UI 风格

产出：

- 一版可演示的数字纪念馆

完成标准：

- 场景浏览和内容阅读形成闭环

## 阶段 4：性能与发布

目标：让项目可以稳定对外访问。

任务：

- 压缩场景资产
- 优化加载策略
- 适配桌面端和基础移动端
- 选择并完成部署

产出：

- 线上可访问版本

完成标准：

- 主要设备上可打开、可浏览、可分享

## 阶段 5：第二阶段能力（暂缓）

候选内容：

- 后台管理
- 登录与权限
- 用户投稿
- 视频内容
- 更丰富的内容编辑能力

## 迭代节奏建议

每次迭代遵循以下模板：

1. 先写本次目标
2. 定义完成标准
3. 只改动相关文件
4. 做最小验证
5. 记录结果与下一步

推荐单次迭代大小：

- 30 分钟到 2 小时
- 只解决一个问题
- 最终产出必须“可看”或“可判断”
