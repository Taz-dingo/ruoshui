# Iteration 004 · `SOG` 直加载 Web MVP

## 目标

基于 `assets/hhuc.sog` 直接启动若水广场首个正式 `Web MVP`，优先回答“现成高质量压缩模型能否直接支撑桌面端纪念展示”。

## 输入

- `assets/hhuc.sog`
- `docs/project/spec.md`
- `docs/project/plan.md`
- `docs/project/tasks.md`
- `docs/project/state.md`

## 设计前提

### Visual Thesis

以梧桐树皮与新叶色为基调，做一块更温暖的纪念展陈幕布，场景本身就是首页。

### Content Plan

- Hero：项目名、项目一句话定位、进入场景
- Support：模型体积、格式、当前加载状态
- Detail：导览镜头与当前相机信息
- Final CTA：继续进入全览或切换策展镜头

### Interaction Thesis

- 加载前先给出纪念语境，而不是技术噪音
- 镜头预设像策展导览一样切换
- 内容层保持轻量，只做提示，不压过场景本身

## 输出

- `web/`：`pnpm + Vite` 前端项目
- `web/public/content/mvp.json`：首版文案、镜头与多版本 `SOG` 元数据
- `web/vite.config.mjs`：将原始版与派生版 `SOG` 统一映射为 `/models/*`
- `outputs/iteration-004-sog-opt/`：首轮交付侧派生 `SOG` 候选

## 验证方式

- 在 `web/` 目录运行 `pnpm install`
- 运行 `pnpm dev`
- 确认同页版本切换可在原始版、`h0`、`opacity01`、`dec75`、`dec50` 之间完成重载
- 运行 `pnpm build`
- 确认所有 `SOG` 版本都能作为构建产物发出，且页面脚本通过构建

## 当前结果

- 已将正式 viewer 路线切到 `PlayCanvas Engine API + gsplat + SOG`
- 已提供全屏场景背景、加载状态、镜头预设和当前相机信息面板
- 已避免复制模型文件：开发与构建均直接复用仓库根目录资产与 `outputs/iteration-004-sog-opt/` 派生资产
- 已新增同页多版本 compare 交互：版本按钮、版本元数据面板、切换时重挂载运行时
- 当前默认载入版本已切回 `hhuc-original.sog`，优先把满血原版作为画质与性能基线
- 已将原先不可见的动态分辨率策略改为前端可调的渲染清晰度滑块，并通过本地存储记住用户选择
- 已重排桌面端信息层级：把显示控制上提到版本区下方，并让左右侧栏在视口内独立滚动，避免关键控件掉到首屏以下
- 已调整模型切换交互：切换 `SOG` 版本时默认保留用户当前视角，而不是强制跳回预设镜头
- 已收紧左侧 `hero` 信息块的尺寸、标题和按钮规格，减少其对场景首屏的占用
- 已移除毛玻璃调节按钮，当前界面只保留与模型对比直接相关的控制
- 已进一步压缩右侧信息层：`MVP 取舍` 改为轻量摘要块，`当前聚焦` 改为更短的说明卡，减少对场景的干扰
- 已按当前取舍继续收口：移除 `MVP 取舍` 与 `当前聚焦` 两个次要面板，只保留版本、清晰度、镜头与相机信息
- 已继续精简左侧与版本区信息：`hero` 只保留标题、短说明和两个入口按钮；版本卡只保留名称与核心体积信息；左右栏宽度与面板边缘已统一对齐
- 已进一步按单屏工具轨重整 UI：左右两侧收敛为对齐的窄栏，弱化卡片感与过重阴影，并缩短状态与清晰度文案，让首屏更明确地服务场景本身
- 已将整体视觉从冷色科技感转向梧桐灵感：以暖树皮色和嫩叶绿替换青蓝高亮，并继续压缩右侧“模型版本”区的高度
- 已移除含义较弱的“记忆锚点”列表，改为实时显示当前相机的位置、目标、距离与朝向
- 已把右侧操作层改为默认折叠的按需展开面板，并将模型版本切成两列，减少首屏 UI 占用
- 已继续优化默认交互：左右信息层都改为默认透明、仅保留文字；悬停后再显出整块卡片，同时移除相机拖动时对 UI 毛玻璃强度的动态改动，固定为常态视觉
- 已继续校正左右首屏权重：适当放大右侧控制栏宽度与内边距，避免在透明待机态下显得比左侧过轻
- 已继续放大右侧控制栏，并去掉悬停时的横向滑动位移，让其显现更稳、更接近固定工具栏
- 已开始首轮前端性能收口：切换模型时主动清理旧运行时挂到 `window` / `canvas` 的事件监听，并降低相机信息面板的 DOM 刷新频率，避免交互越久越重
- 已追加静态场景按需渲染：场景启动或交互后先短暂保持连续绘制，待镜头静止后再自动收敛，只在镜头、分辨率、窗口尺寸或模型状态变化时重新唤醒，降低静止时的 GPU 占用
- 已补一个只在开发环境显示的轻量性能角标：显示 FPS、平均帧时、当前渲染状态与渲染比例，便于后续每轮优化做主观与客观对照
- 已继续压低交互期主线程开销：为 orbit 控制与相机更新复用向量缓存，减少拖拽、缩放与镜头阻尼过程中的临时对象分配和 GC 压力
- 已按 `PlayCanvas` 文档建议关闭后备缓冲抗锯齿，并为图形上下文声明 `high-performance` 偏好，继续压低浏览器端绘制开销
- 已生成一版实验性 `LOD streaming` 资产：用 `hhuc-h0 / opacity01 / dec75 / dec50` 四档 `SOG` 打包出 `lod-meta.json` 与节点资源，并已接入 viewer 作为 `LOD 流式` 对比版本，便于验证“总包更大但首屏更轻”的路线是否成立
- 已补页面失焦/恢复的生命周期收口：切后台时主动结束交互并暂停渲染，切回前台时再短暂唤醒渲染，避免浏览器切页后状态残留导致的性能回落
- 已为 `LOD streaming` 补首轮节奏调优：放宽视角/距离触发阈值、延长节点冷却，并允许轻度 `underfill`，优先减少拖拽与缩放过程中的频繁节点切换抖动
- 已补模型级轻量性能统计：当前版本面板会记录最近一次 `加载耗时 / 首帧耗时 / 漫游平均帧时`，便于切换原始版、轻量版与 `LOD` 版时做更可复现的主观对照
- 已补一套固定相机对比轨迹：`结构巡航 / 侧向扫视 / 近景逼近` 三条路径均可一键重复播放，用统一视角路径比较首屏细化、结构稳定度与移动期抖动
- 已把空闲态从“仅停渲染”进一步收口为“主循环睡眠 + 交互唤醒”，优先压低长时间打开页面后可能出现的持续 `tick` 开销与热衰减风险
- 已把 `漫游` 指标从生命周期平均改为“最近一轮移动会话”的平均 / 峰值帧时，避免旧样本把后续掉速稀释掉，更方便判断“刚进来顺、过一会卡”的时间型问题
- 已为对比轨迹补自动性能记录：每次播放后都会自动记下版本、清晰度、时间戳、首帧耗时与最近一轮移动的平均 / 峰值帧时，减少手记误差
- 已补“当前轨迹 × 全版本”一键标准测试：会按统一轨迹顺序重载并跑完所有模型版本，把结果连续写入自动记录区，方便直接横向对比
- 已将版本切换从“重建整套 `PlayCanvas Application`”改为“复用单一运行时并只替换 `gsplat` 资产”，避免批量标准测试触发过多 `WebGL context` 警告
- 已把标准测试 telemetry 扩成“逐帧 trace + 长任务 + 模型资源请求”三层记录：每轮轨迹现在会保留帧时、所处 step、相机位置/目标、长任务窗口与 `/models/` 资源补载时序，便于回溯“具体哪个视角开始卡”
- 已在右侧轨迹面板补“标准测试分析”区：同一批次会给出版本排行榜、每版最差 step，以及跨版本最高峰卡顿热点，帮助快速锁定是某一段镜头还是某一版资产更容易抖动
- 已补导出链路：最新一轮标准测试可直接复制摘要、复制 JSON、下载 JSON，并同时挂到 `window.__ruoshuiPerf.latest()` / `window.__ruoshuiPerf.history()`，方便后续把数据交给我继续分析
- 已补一个更轻的单版本入口：现在可直接跑“当前轨迹 × 当前版本”，会沿用同一套自动记录、分析与 JSON 导出，方便只验证 `LOD 流式` 这类定向改动，而不必每次重跑全版本
- 已把单版本入口默认扩成重复 `3` 次，优先压低单次跑分受缓存、热状态和浏览器瞬时波动影响，方便更稳定地比较 `LOD 流式` 与原始版
- 已补一条本地自动化 benchmark 脚本：`web/scripts/benchmark-auto.mjs` 会自动构建、拉起 `vite preview`、打开浏览器、逐个运行指定轨迹 / 版本 / 重复次数，并把每次导出的 JSON 与汇总表直接写入 `outputs/iteration-004-web-benchmarks/`，不再需要手动下载多个文件
- 自动化脚本当前优先使用系统 `Google Chrome`，且默认以可见窗口运行；这是因为这类 `WebGL / gsplat` benchmark 在 `headless` 模式下不够稳定，容易引入额外偏差。如需指定浏览器路径，可传 `--browser-path` 或环境变量 `RUOSHUI_BENCHMARK_BROWSER`
- 已给 orbit 相机补俯仰下限，禁止视角翻到模型底部；手动旋转、预设镜头和切模型后的视角恢复都会统一钳在可接受仰角之上
- 已开始验证“低视角高密度 splat 突增”假设：当前先对 `LOD 流式` 版加入低机位 / 近景风险窗口记录与短时保活渲染，用于观察热点是否稳定集中在低俯仰、近距离视角；此前直接动态改 `LOD` 参数的尝试已回滚，因为会触发底层 `gsplat` 运行时错误
- 已把这轮风险窗口记录写进标准测试 trace：导出的 `JSON` 现在会额外记录 `trace.lodWarmups`，后续可继续对照“风险窗口是否命中热点视角”与“卡顿是否更偏向资源补载还是视角相关排序 / GPU 峰值”
- 已开始做前端代码整理：`web/src/main.js` 已迁到 `TypeScript` 入口，并先拆出 `config / types / benchmark analysis / render-scale / utils` 几个纯逻辑模块，避免继续把 UI、运行时、数学工具和性能分析都堆在一个文件里
- 已补 `web/tsconfig.json` 与 `pnpm typecheck`，当前这批基础拆分在本地已通过 `typecheck + build`，后续可以继续按“先抽纯函数、再拆运行时和 UI 编排”的节奏收口
- 已把 UI 壳子迁到 `React + TSX`：当前由 `web/src/main.tsx` 挂 `React root`，`web/src/App.tsx` 负责渲染页面骨架，旧的 PlayCanvas / benchmark 运行时则先保留在 `web/src/viewer.ts` 中，通过既有 DOM id 挂接到 React 渲染出的节点
- 这轮迁移的目的不是立即重写整套 viewer 逻辑，而是先拿掉 `innerHTML` 模板拼装和大段原生 DOM 壳子代码，为后续继续拆 `runtime / orbit / benchmark UI / inspector state` 建立更干净的边界
- 已修复 React 壳子迁移后的黑屏回归：首屏会先同步挂出场景容器，再显式同步 `canvas` CSS 尺寸与 `graphicsDevice` backing resolution，避免出现 DOM 已在但底层绘制分辨率仍是 `0x0` 的情况
- 已修复窗口变化后“UI 会缩放、场景不跟随”的问题：viewer 现在改为监听场景容器尺寸并通过 `ResizeObserver` 同步 `canvas` 与渲染分辨率，而不是继续读取被旧像素尺寸锁住的 `canvas.clientWidth / clientHeight`
- 已补 orbit 相机与目标点的高度下界：当前统一限制 `camera position.y >= 0` 且 `target.y >= 0`，避免镜头落到场景下方或继续看向地下

## 补充实验：`SOG` 派生轻量版本

在不能重训第三方模型的前提下，已用 `@playcanvas/splat-transform` 对 `assets/hhuc.sog` 做首轮交付侧派生实验，输出位于 `outputs/iteration-004-sog-opt/`。

- 原始 `hhuc.sog`：约 `27.13 MiB`，`1,868,855` splats
- `hhuc-h0.sog`：约 `21.62 MiB`，保留 `100%` splats，仅去掉高阶 SH
- `hhuc-h0-opacity01.sog`：约 `18.58 MiB`，保留约 `84.8%` splats
- `hhuc-h0-dec75.sog`：约 `16.70 MiB`，保留 `75%` splats
- `hhuc-h0-dec50.sog`：约 `11.45 MiB`，保留 `50%` splats

当前最值得优先主观对比的两版是：

- `hhuc-h0.sog`：低风险、轻度减重
- `hhuc-h0-dec75.sog`：更可能带来明显性能改善

其中 compare 页当前默认先落在 `hhuc-original.sog`，优先把满血原版作为当前体验与性能基线；`hhuc-h0-dec75.sog` 仍保留为最值得重点对比的轻量候选版本。

## 风险与问题

- 当前仍未做真实线上部署，首轮只保证本地开发与构建链路
- 镜头预设是基于现有包围盒做的首轮人工设置，后续仍应根据真实观感微调
- 若后续还要回到“记忆点位”叙事，应优先直接做 3D 场景内热点，而不是恢复当前这类面板列表
- 界面玻璃感当前仍是前端固定预设档位，后续若需要更细控制，可再改为连续滑杆
- 渲染清晰度当前已改为手动锁定比例；后续若需要“自动 / 手动”双模式，再补一个切换入口

## 下一步

- 基于 compare 页的主观体验，先从 `dec75` 与 `h0` 中选出正式默认版本
- 若 `dec75` 质量可接受，继续围绕它做首屏加载与交互细化
- 若切换体验稳定，再决定是否追加 `SOG` 侧进一步裁切、分层或发布优化
