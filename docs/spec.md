# 项目 Spec

## 1. 项目名称

暂定名：`Ruoshui`

可理解为一个“老校区空中数字纪念馆”。

## 2. 背景

项目起源于对常州老校区的纪念。用户在大学前两年曾在该校区生活，后续校区将被废弃或迁移，因此希望通过数字化方式保留其空间记忆。

与普通校园导览不同，这个项目的核心不是功能复杂度，而是“重新看到那个地方”的沉浸感和纪念价值。

## 3. 项目目标

### 3.1 核心目标

- 基于现有航拍五向图，尝试重建老校区的空中版 3DGS 场景
- 以 Web 作为第一发布端，保证尽可能低门槛访问
- 提供高质量空中浏览体验，支持预设机位、导览与少量热点内容
- 用最小可行范围验证：素材是否足够、场景是否可看、页面是否可发布

### 3.2 非目标（第一阶段不做）

- 不优先做开放式用户发帖社区
- 不优先做复杂后端系统
- 不优先做小程序适配
- 不强行追求地面步行漫游体验
- 不在第一阶段解决完整 UGC 审核体系

## 4. 用户与场景

### 4.1 目标用户

- 自己
- 曾在老校区生活过的同学、校友
- 对校园记忆、数字纪念项目感兴趣的人

### 4.2 核心使用场景

- 用户打开网页，进入老校区空中场景
- 用户通过镜头导览浏览校园主要区域
- 用户在关键点位查看配套图文记忆
- 用户分享某个固定视角或故事点位给他人

## 5. 产品范围

### 5.1 第一版必须有

- 一个可在 Web 打开的 3DGS 场景页
- 场景加载状态与错误兜底
- 若干预设镜头 / 导览路径
- 若干热点记忆点位
- 每个点位的图文内容展示层
- 项目说明页

### 5.2 第一版可以有

- 背景音乐开关
- 时间轴式地点导航
- 镜头过渡动画
- 简单移动端适配

### 5.3 后续版本再做

- 登录系统
- 内容后台
- 点位 CMS
- 用户投稿
- 视频内容
- 小程序版本

## 6. 资产与技术假设

### 6.1 已知输入

- 一批校区航拍五向图
- 重心是空中展示，不做地面沉浸漫游

### 6.2 关键假设

- 五向图素材足以支持“空中版” 3DGS 重建
- 场景质量优先于复杂交互
- Web 端需要压缩、裁切、分级加载，否则体验会受影响

### 6.3 最大风险

- 素材视角不足，导致重建缺面、漂浮、噪声严重
- 3DGS 原始资产过大，Web 首屏过慢
- 移动端设备性能不足
- 空中视角好看，但叙事性不足

## 7. 技术路线（当前建议）

### 7.1 资产生产链路

候选流程：

1. 素材清理与筛选
2. 相机位姿恢复 / 稀疏重建
3. 3DGS 训练
4. 资产裁切、清理、压缩
5. Web Viewer 集成

### 7.2 技术栈建议

- 前端框架：`Next.js`
- 3D 展示：优先接入成熟的 `3DGS Web Viewer` 能力
- 页面结构：前端负责外层叙事、导航、信息层
- 内容层：第一版先用本地 JSON 或静态内容维护
- 部署：优先静态站点或轻量 Node 部署

### 7.3 数据结构（第一版）

第一版尽量无后端，内容可先静态化：

- `scene`
  - `id`
  - `title`
  - `description`
  - `assetUrl`
  - `previewImage`
- `cameraPreset`
  - `id`
  - `name`
  - `position`
  - `target`
- `memoryPoint`
  - `id`
  - `title`
  - `summary`
  - `position`
  - `cover`
  - `body`

## 8. 页面信息架构

### 8.1 首页

- 项目标题
- 背景介绍
- 进入场景按钮
- 技术与纪念说明

### 8.2 场景页

- 3DGS 主视图
- 加载进度
- 镜头预设切换
- 热点入口
- 信息面板

### 8.3 故事 / 点位面板

- 标题
- 地点说明
- 图文内容
- 返回场景

## 9. 验收标准（第一版）

满足以下条件视为第一版方向成立：

- 能训练出可辨识的老校区空中 3DGS 结果
- 能在浏览器中完成基本展示
- 用户能进入、观看、切换视角、查看热点
- 整体体验不依赖安装原生客户端

## 10. 开发原则

- 先验证风险，再扩展功能
- 每次迭代只解决一个主要问题
- 文档先行，结果可验证
- 能静态化解决的问题，先不要引入后端
- 不为未来功能提前设计过多复杂结构
